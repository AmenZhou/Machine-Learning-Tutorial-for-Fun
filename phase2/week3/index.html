📖 What is Multiple Regression?

Plain-English Hook

Week 1: one dial (servers → bill). Week 3: five dials. Each feature pulls the prediction up or down by a different amount. The model finds the best weight for every dial at once.

From 1 Feature to 5

In Week 1, linear regression drew one line through a scatter plot: bill = slope x servers + intercept. With multiple regression, we extend that idea to handle five features simultaneously:

bill = a·servers + b·db_reads + c·storage + d·api_calls + e·deploys + intercept
Each coloured letter is a coefficient — how much the bill changes for each unit increase in that feature.
Connection to Week 1: In Week 1, slope = how much the bill changes per additional server. Now there are 5 slopes, one per feature — each measures the marginal cost of that resource.

Why More Features Help

Your AWS bill doesn’t just depend on server count. DB reads, storage, API traffic, and deploy frequency all add cost. A one-feature model ignores up to 80% of the signal — like trying to guess a recipe’s calories from the flour alone.

How It Still Works

Under the hood it’s the same math from Week 1 — Ordinary Least Squares (OLS). The goal is still to minimise the sum of squared errors between predicted and actual bills. The only difference: instead of solving for one slope, we solve for five simultaneously using matrix algebra.

Week 1
Minimise Σ(y − ŷ)²
1 slope, 1 intercept
Solve: 2 equations
Week 3
Minimise Σ(y − ŷ)²
5 slopes, 1 intercept
Solve: XᵀXβ = Xᵀy
New term — Multicollinearity: When two features are highly correlated with each other (e.g., servers and db_reads both go up together), the model struggles to separate their individual contributions. We’ll explore this in Tab 4.

📊 Feature Importance


  

    
Engineering Question

    
Which monitoring metric predicts the bill most reliably? Not the one with the biggest raw coefficient —
    the one that explains the most variance when we account for each feature’s scale.

  


  

    
Normalised Importance (Standardised Coefficients)

    
    
    

    

      Why normalise?
      Servers has the highest raw coefficient (~$200/unit) but api_calls varies the most (10 → 46),
      so its actual impact on the bill is comparable. Normalised importance = |coef| × std(feature) / std(bill).
      This accounts for scale differences so you can compare apples to apples.

Before You Model

    
Before building the model, check: which features are highly correlated with the bill? And with each other?
    Correlated features can confuse the model — this is called multicollinearity.

  


  

    
Pearson Correlation Matrix (5 features + bill)

    
    
    

    

      Multicollinearity note:
      All 5 features are strongly correlated with the bill — that’s good, they carry useful signal.
      They’re also correlated with each other (because the company grew steadily). This is
      multicollinearity: a warning sign for very large datasets or noisy data,
      but less of an issue here with clean, structured growth data.

🎓 Quiz — Multiple Regression


  

    
Test Your Understanding

    
Three questions. Think it through before clicking — there’s no timer, no penalty for reading carefully.

  


  

    
Q1. You add a 6th feature: random_number. R² goes from 0.998 to 0.999. Should you keep it?

    
    
    
    

      Correct! A random feature will almost always cause a tiny R² increase on training data (overfitting).
      Use adjusted R² or hold-out validation to test whether a new feature really earns its place.
    

  


  

    
Q2. Feature importance shows servers = 0.95, db_reads = 0.12. What does this mean?

    
    
    
    

      Correct! Normalised importance (standardised coefficient) measures how much each feature’s natural variation
      drives the bill. Servers dominates; db_reads contributes but is secondary. “Remove it” is too strong —
      it may still help in other datasets or with different variance patterns.
    

  


  

    
Q3. Multiple regression finds: bill = 200·servers + 150·db + 80·storage + 30·api + 100·deploys + 180. What is 180?

    
    
    
    

      Correct! The intercept is the model’s baseline prediction when every feature equals zero.
      In practice it represents fixed costs — account overhead, minimum charges — that exist regardless
      of compute usage.
